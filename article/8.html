Context Engineering: Why Your Agent's Memory Architecture Is Wrong by Default

Samer Alkhodary
•8 min read•5 topics

Most agent developers treat the context window as an append-only database and reach for a vector store when the agent starts forgetting things. The correct mental model is RAM: fast, finite working memory that needs deliberate tiers below it — and the write problem is harder than the read problem.

#AI Agents#Context Engineering#Memory#Multi-Agent#RAG

TL;DR

  • The context window is working memory, not storage. Every token you put in it competes with every other token. Appending indefinitely degrades performance before you hit the hard ceiling.
  • There are three tiers that survive production: in-context working memory, external retrieval (warm), and persistent structured state (cold). Each has a different read/write contract.
  • Deciding what to write to persistent memory is significantly harder than deciding how to read from it. Most frameworks paper over this with naive summarization.
  • Letta's git-backed context repositories are an interesting solution to a specific hard problem: concurrent writes from multiple subagents. Worth understanding even if you're not using Letta.

Why this matters

The "agent forgot what I told it three turns ago" problem is real, and the instinct to solve it by adding a vector database is understandable but often wrong. A vector store solves the retrieval problem. It does nothing about the harder question of whether the agent wrote anything worth retrieving in the first place. Engineers who have shipped multi-session agents will recognize the pattern: elaborate retrieval infrastructure on top of shallow memory writes, with the system confidently surfacing the wrong facts at query time. Before reaching for more infrastructure, it's worth getting the mental model right.

What's actually going on

§The RAM metaphor, and why it matters

The context window is bounded working memory. Tokens at different positions have different effective weights — very long contexts degrade recall on items in the middle 3, and every irrelevant token you include is competing with the tokens that actually matter. Treating the context window as a database that you append to indefinitely works fine in demos and falls apart in production.

The right analogy is RAM in a program with a fixed heap. You have fast access to everything currently in it, but you cannot fit the whole application state there at once, writing to it has side effects on everything that reads it, and you need a deliberate architecture for what gets paged out and when.

§The three tiers

Tier 1: In-context working memory. This is everything currently in the prompt — the system prompt, conversation history, injected structured state, and any retrieved snippets. It's the only memory the model actually sees. It is fast and always relevant to the current inference, but it is finite and ephemeral. When the session ends, it's gone unless something explicitly writes it down.

Tier 2: External retrieval (warm memory). Vector stores, keyword indexes, embedding search — all of this lives here. You write facts to it and read them back via semantic or keyword query. The read latency is low enough to inject results into context before inference. The problem is that retrieval is a probabilistic operation: the AMA-Bench evaluation found that "similarity-based retrieval is unreliable," with HippoRAG2 dropping 43.2% accuracy when moving from constructed memory (where the correct needle is guaranteed present) to end-to-end retrieval 3. Retrieval infrastructure is not a memory solution; it is a recall mechanism on top of a memory solution.

Tier 3: Persistent structured state (cold memory). This is the agent's long-term knowledge — a typed schema of what it knows about a user, task, or domain. The OpenAI context personalization cookbook 4 implements this as a structured profile (typed fields like seat_preference, loyalty_status) plus a set of timestamped notes. Reads are explicit: you render the state into the system prompt at session start. Writes are deliberate: a consolidation step at the end of each session decides what gets promoted from session memory to global memory, what gets deduplicated, and what gets dropped. The cookbook's consolidation prompt includes an explicit instruction: "Do NOT invent new facts. Only use what appears in the input notes" 4 — a guard against the model hallucinating memories during summarization.

§The write problem is harder

Most of the tooling and literature focuses on retrieval — embedding models, vector store backends, rerankers. The write side gets much less attention, but it is where most production memory failures originate.

The questions on the write side:

  • When do you write? (After every turn? At session end? When confidence exceeds a threshold?)
  • What do you write? (The raw exchange? A model-generated summary? Extracted structured facts?)
  • How do you handle contradiction? (A new note that conflicts with an existing one needs resolution logic, not just appending.)
  • How do you handle forgetting? (Notes that are only relevant to one session should be dropped, not promoted to global memory.)

The OpenAI cookbook's session-end consolidation addresses all four explicitly: it deduplicates semantically equivalent notes, resolves conflicts by keeping the most recent, and explicitly drops ephemeral session-only observations 4. This is not exotic — it is careful bookkeeping. The point is that someone has to implement this logic, and "dump it all in the vector store" is not an answer.

AMA-Bench found that "varying the memory architecture induces significantly higher variance" than model size, with score ranges reaching 0.45 across memory system variants 3. You will get more leverage from better write logic than from a bigger model.

§Filesystem-as-memory

One finding that surprises people: storing agent memory as plain files on a filesystem performs competitively on long-horizon benchmarks. Mem0's 2026 State of AI Agent Memory report 2 includes LoCoMo and LongMemEval benchmark comparisons, though I want to be direct about the source: this is a vendor-published report from a company that builds memory infrastructure. Their own algorithm scores 91.6 on LoCoMo. I could not verify a specific "filesystem at ~74%" figure I had seen attributed to this report — the fetched content did not surface it. Take any comparison numbers from this source as a starting point for your own evaluation, not as ground truth.

The intuition behind why filesystem-based memory is competitive is sound regardless: plain text files give the agent (or a retrieval system) structured navigation, token-efficient frontmatter for quick scanning, and a human-readable representation that is easy to inspect and debug. Letta's approach 1 makes this explicit: their context repositories use a file tree where the directory hierarchy and file names act as navigational signals, and a system/ directory designates which files are always fully loaded into the system prompt. You do not need semantic embedding search to find a file called user/preferences.md.

§Letta's git-backed context repositories

Letta's context repository design 1 is interesting for a specific reason that has nothing to do with whether you use Letta: it addresses concurrent writes from multiple subagents, which is a genuinely hard coordination problem.

In a single-agent, single-session setup, memory writes are sequential and the ordering is clear. In a multi-agent system where subagents are running in parallel, you have the standard distributed systems problem of concurrent writes to shared state. Two subagents processing different parts of a task simultaneously both want to update the shared memory. Without coordination, you get last-write-wins corruption or conflicts that silently discard one agent's output.

Letta's solution: give each subagent an isolated git worktree. Each subagent writes to its own working copy of the memory repository. When they finish, their changes merge back through git's conflict resolution. Every change is versioned with a commit message 1.

What the git metaphor buys you:

  • Isolation: subagents can't corrupt each other's in-progress writes
  • Auditability: you have a full history of every memory change, who made it, and when
  • Rollback: if a subagent writes bad information, you can revert to a known good state
  • Conflict visibility: when two subagents contradict each other, git surfaces the conflict explicitly rather than silently dropping one

This is not the right architecture for every use case — a single-agent chatbot does not need worktree isolation. But if you are building multi-agent systems where subagents run concurrently and share memory state, this pattern solves a real problem that most frameworks either ignore or handle with coarse-grained locking.

When do you actually need a vector store

A vector store makes sense when:

  • Your memory corpus is large enough that loading it all into context is impractical (thousands of notes, not dozens)
  • The relevant memories are genuinely unpredictable from session to session, so you cannot pre-select what to inject
  • You have invested in write logic that produces retrieval-worthy chunks (not raw conversation transcripts)

A vector store is cargo cult when:

  • You have fewer than a few hundred memory entries — a simple prefix scan or keyword index is faster, cheaper, and more debuggable
  • Your retrieval is hitting the same notes every session — those should be in the structured profile, not in a retrieval corpus
  • You haven't defined your write logic yet — adding semantic search on top of unstructured notes does not fix the fundamental problem

The AMA-Bench results reinforce this: "the long context baseline is consistently strong and often achieves the best performance, whereas existing memory systems exhibit large variance" 3. For many tasks, carefully curating what goes into context beats a retrieval layer over everything. The baseline is harder to beat than the tooling ecosystem implies.

Caveats and open questions

The benchmark landscape is messy. LoCoMo, LongMemEval, AMA-Bench, BEAM — each measures different things with different methodologies. The Mem0 report 2 is vendor-sourced and compares against their own prior algorithm, not against a neutral independent baseline. AMA-Bench 3 is academic and more credible, but it covers agentic trajectories that may not match your use case. No single benchmark tells you how memory will perform on your workload.

The write logic problem is unsolved at the ecosystem level. Most frameworks (LangGraph, LangChain, CrewAI) give you retrieval infrastructure and leave write logic to you. Some defaults (summarize the last N turns) are actively harmful for long-horizon tasks where subtle facts get dropped. This is an area where you should have opinions rather than accepting framework defaults.

Git-backed memory has real operational costs. Worktrees, merge conflicts, and git operations add latency and complexity. Letta's approach 1 makes sense when concurrent writes are a genuine bottleneck. Adopting it when you have one agent with sequential memory writes is over-engineering.

Memory and privacy are in tension. The OpenAI cookbook 4 explicitly bans storing "passport numbers, payment details, SSNs, full DOB, addresses" in the memory notes. This is table stakes, not a complete answer. Long-lived memory stores that accumulate facts about users over months become a significant privacy surface — one that most memory library documentation addresses superficially if at all.
