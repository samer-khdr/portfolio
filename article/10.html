MCP vs Skills: Two Different Layers for Extending Agent Capabilities

Samer Alkhodary
•7 min read•5 topics

MCP and Anthropic's Agent Skills both extend what an agent can do, but they operate at different layers and solve different problems. This post breaks down the technical architecture of each, where they overlap, where they are complementary, and what signals exist about their respective trajectories.

#MCP#Agent Skills#AI Agents#Tooling#Architecture

TL;DR

  • MCP is a wire protocol: it standardizes how any client connects to external tools, data sources, and services. Think of it as the integration layer.
  • Skills are context-loaded capability packages: a 
    SKILL.md
    plus optional scripts and docs that teach an agent how to do something, loaded progressively into the context window to manage token cost.
  • They are not competing alternatives. Anthropic's own framing is explicit: MCP = connectivity, Skills = expertise. A single skill can orchestrate multiple MCP servers 4.
  • MCP is now a vendor-neutral standard under the Linux Foundation's Agentic AI Foundation. Skills remain an Anthropic-specific mechanism. These are on different governance and portability trajectories.

Why this matters

If you are building agent systems today, you have to choose what to invest in. Do you build MCP servers to expose your internal tools? Do you package reusable workflows as Skills? Both? The answer depends on understanding what each one actually is — not the marketing framing, but the technical layer it operates at and the problem it is designed to solve.

What MCP actually is

MCP (Model Context Protocol) is an open-source client-server protocol for connecting AI applications to external tools, data, and services 1. It defines three primitives:

  • Tools: executable functions the model can invoke (a database query, a file write, an API call)
  • Resources: data the server exposes for the model to read (files, records, live content)
  • Prompts: reusable prompt templates the server can provide

An MCP server is a process that implements this protocol and exposes some set of capabilities. The client — which could be Claude Code, VS Code Copilot, Cursor, or anything else that speaks MCP — connects to those servers and makes their capabilities available to the model 1.

The protocol's architecture is small: clients maintain one-to-one connections with servers, and the host (the application, e.g., Claude Code) manages those connections and decides what gets exposed to the model 1. In Claude Code specifically, you configure MCP servers with 

claude mcp add
, and from there the model can call into Jira, Sentry, PostgreSQL, or whatever you have wired up 8.

MCP's scope is broad: it handles the connection, the capability negotiation, and the invocation protocol, but it says nothing about how to accomplish complex multi-step workflows. It tells the model what tools exist; it does not teach the model expert-level usage patterns for those tools.

What Skills actually are

A Skill is a directory. Specifically:

.claude/skills/
  comparable-company-analysis/
    SKILL.md          # required: YAML frontmatter + instructions
    templates/        # optional extra files
    scripts/          # optional runnable code

The 

SKILL.md
has a YAML frontmatter block with a 
name
and 
description
field. That metadata — roughly 100 tokens — is what the model loads at startup. When a user request matches a skill's domain, the model loads the full 
SKILL.md
(under 5k tokens). If that 
SKILL.md
references additional files, those are loaded on-demand via bash 3.

This three-level progressive disclosure (L1: metadata always loaded; L2: full SKILL.md on trigger; L3: referenced files on-demand) is the key design decision. The token cost at idle is negligible; the full context is only paid when the skill is actually relevant 3.

Skills live at 

~/.claude/skills/
(personal, available across projects) or 
.claude/skills/
(project-local) 3. The pre-built skills from Anthropic cover document formats: PowerPoint generation, Excel manipulation, Word and PDF handling 3.

The core proposition is different from MCP's: a Skill packages workflow expertise, not connectivity. Anthropic frames Skills as teaching agents "more complex workflows that involve external tools and software," specifically as a way to complement MCP 2.

Where they differ, concretely

Protocol vs. context injection. MCP requires a running server process that speaks the protocol. The client connects at runtime, discovers capabilities, and the model can invoke them as tool calls. A Skill is Markdown read from the filesystem. No server, no protocol negotiation, no network traffic.

Token cost. Willison cited GitHub's official MCP server as consuming tens of thousands of tokens when its full capability set is loaded into context 5. A Skill's startup cost is a few dozen tokens — the L1 metadata — scaling to thousands only when actively triggered 5. This is a meaningful difference if you are managing context budgets across many capabilities.

Portability. MCP is a vendor-neutral standard. Any client that implements the protocol can connect to any MCP server 1. Skills are currently an Anthropic mechanism — 

SKILL.md
in 
.claude/skills/
is a Claude Code convention. Willison noted in October 2025 that Skills could work with Codex CLI and Gemini CLI; I could not independently verify cross-vendor Skills adoption from the sources in this article, so treat that as Willison's claim at that time, not an established fact 5.

Scope of capability. MCP is good at: "expose this tool so the model can call it." Skills are good at: "teach the model the correct multi-step workflow for this domain." These are genuinely different problems. A Skills-based Comparable Company Analysis workflow can coordinate three MCP-connected financial data sources and know the correct sequencing, fallback logic, and output format — none of which the MCP servers themselves encode 4.

How they fit together

Anthropic's framing from December 2025 is the clearest statement of the intended relationship: MCP provides connectivity, Skills provide expertise 4. A single skill can orchestrate multiple MCP servers. A single MCP server can support dozens of different skills 4. They are complementary layers, not competing mechanisms.

In Claude Code, both appear as listed extension mechanisms: "Extend Claude Code with skills, connect to external services with MCP" 8. Claude Code also documents an 

alwaysLoad
flag for MCP servers (requires Claude Code v2.1.121+) that forces server tools into context at session start 8.

A practical heuristic: if you are wiring up a new data source or service so that the model can call into it, that is an MCP server. If you are encoding a reusable workflow pattern or domain expertise that the model should execute when it encounters a particular class of task, that is a Skill.

Where each is heading (inference, not prediction)

On the MCP side: the December 2025 donation to the Agentic AI Foundation — a directed fund under the Linux Foundation, co-founded by Anthropic, Block, and OpenAI, with Google, Microsoft, AWS, Cloudflare, and Bloomberg as Platinum members — is a strong signal about trajectory 67. The AAIF governs strategy and budget; technical governance remains via the Steering and Evolution Process 7. When a protocol gets that governance structure, it is being positioned as infrastructure. The implication, as inference: MCP is being positioned as vendor-neutral infrastructure — not something any one vendor owns.

On the Skills side: there is no published roadmap. What is visible from current sources is that Skills are Claude Code-specific, actively developed, and positioned as the mechanism for encoding complex agent workflows. If MCP continues to grow as the integration standard, Skills could evolve as the layer above MCP — the expertise layer that agents load to know how to use what MCP exposes. But that is a reading of the architecture and current positioning, not a statement about Anthropic's plans.

Cross-vendor Skills adoption remains an open question. The Willison claim about Codex CLI and Gemini CLI compatibility would, if true, make Skills a more portable mechanism than they currently appear — but this article cannot verify it from the available sources.

Caveats and open questions

No head-to-head production benchmarks exist in these sources. The Willison comparison — tens of thousands of tokens for a large MCP server vs. a few dozen for a Skill's startup metadata — is the only concrete comparative number available 5. It is directionally meaningful but not a controlled study. Your actual token costs depend on how many MCP servers you run, what their capability sets look like, and how many Skills are idle vs. triggered at any given time.

Skills portability is unverified. The claim that Skills work with non-Claude clients is from a single October 2025 blog post 5. If cross-vendor adoption is relevant to your evaluation, verify against current documentation for each client.

Future direction is inference. Neither Anthropic nor the MCP project has published a roadmap describing how these two mechanisms will evolve relative to each other. Everything in the "where each is heading" section is interpretation of observable signals: the AAIF governance structure, the complementary positioning language, the separate development tracks in Claude Code. Treat it as a working hypothesis, not a product announcement.
