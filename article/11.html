From Jira Ticket to Deploy: A Working Claude Code Pipeline

Samer Alkhodary
•10 min read•5 topics

Vendor demos show Claude writing code in isolation. This article wires the full loop — Jira intake, spec, implementation, tests, PR, security review, deploy — using documented Claude Code primitives (subagents, hooks, MCP servers, claude-code-action@v1). The architectural insight is where the human gates go and why they matter, not the individual pieces.

#Claude Code#MCP#Subagents#GitHub Actions#CI/CD

TL;DR

  • Three things converged in 2026 that make a Jira-to-deploy loop tractable without custom orchestration: the Atlassian MCP server went GA with OAuth 2.1 2, 
    claude-code-action@v1
    reached stable 10, and Claude Code's subagent model gives you isolated context windows per stage by default 3.
  • The right shape is seven sequential subagents — not one sprawling session, and not an agent team. Each subagent has a narrow scope and a structured hand-off to the next.
  • Hooks, not prompts, are the gate mechanism. 
    SubagentStop
    blocks the pipeline until a human approves the plan. 
    PreToolUse
    blocks the deploy command until a human confirms. Asking Claude to ask permission is not a gate.
  • Security is a stage, not an afterthought. 
    /security-review
    is a built-in Skill that runs cheap 67. 
    /ultrareview
    is the deep multi-agent pass at $5–$20 per run after three free 8.

Why this matters

The demo circuit shows Claude writing a feature in a clean repo. The seams of a real pipeline — ticket intake, test execution, PR review, deploy approval, security audit — are where engineering judgment actually lives. If you are evaluating whether to build production infrastructure on Claude Code, the seams matter more than the happy path. This post wires the full loop using documented primitives, with explicit notes on what is stable versus experimental and where the architecture has to compose rather than integrate.

What actually changed in 2026

Two integrations that previously carried real risk are now stable.

Atlassian MCP server (GA February 4, 2026). Before GA you wrote a custom Jira REST wrapper with Personal Access Token auth, managed token refresh, and pinned an API version. The hosted MCP server now handles OAuth 2.1 out of the box and supports thirteen clients including Claude Code 2. There is no local subprocess to manage — you connect to a remote endpoint via 

claude mcp add --transport http atlassian https://mcp.atlassian.com/v1/mcp
. The remote-only model is an operational change worth understanding 1. No npm package to version-pin, no local process supervisor, but you do have to handle OAuth scope configuration on the Atlassian side.

claude-code-action@v1
(GA). Before stable, the GitHub Actions integration was a beta with documented breaking-change risk. @v1 is now stable 10. The claude_args parameter accepts real flags including --max-turns, --model, --mcp-config, --allowedTools, --debug, and --append-system-prompt 10. Wiring this into a production CI pipeline is now a reasonable call.

Subagents have stable semantics. A subagent runs in its own context window with its own system prompt and its own tool allowlist 3. There is no frontmatter field to enable this — isolation is the default. That property is what makes a multi-stage pipeline tractable: the implementer subagent does not carry the ticket-intake conversation history into its decisions.

The seven-stage pipeline

text
Jira ticket
    │
    ▼
[Stage 1] ticket-intake subagent
    │
    ▼
[Stage 2] spec-writer subagent
    │   ◀── HUMAN GATE: plan approval (SubagentStop hook, exit 2)
    ▼
[Stage 3] implementer subagent
    │
    ▼
[Stage 4] test-writer subagent (separate context)
    │
    ▼
[Stage 5] pr-creator subagent
    │   ◀── QUALITY GATE: branch protection (CI must pass)
    ▼
[Stage 6] security-reviewer subagent
    │   /security-review skill + optional /ultrareview
    ▼
[Stage 7] deploy command
        ◀── HUMAN GATE: PreToolUse hook (jq + read confirmation)

All seven stages are subagents. The parent session is the orchestrator: it spawns each subagent in sequence, passes structured JSON between them, and decides whether to proceed.

A single large Skill could describe these steps, but subagents make the pipeline operationally safer. Each stage gets its own context, permissions, and handoff boundary, which reduces context pollution, limits tool access, and makes human approval between stages enforceable.

Stage 1 — Ticket intake

The subagent calls 

mcp__atlassian__issue_read
(the 
mcp__<server>__<tool>
namespace pattern is documented in the hooks reference, where matchers like 
mcp__memory__.*
are shown 5). It returns structured JSON — not prose — that downstream stages consume.

yaml
# .claude/agents/ticket-intake.md
---
name: ticket-intake
description: Read a Jira ticket and return a structured spec object
tools:
  - mcp__atlassian__issue_read
  - mcp__atlassian__search_issues
---
Read the Jira ticket at $TICKET_KEY. Pull any linked issues.
Return JSON with: title, acceptance_criteria (array), linked_tickets,
labels, and known_constraints. Do not write code or modify files.

Scope this subagent to read-only. The Atlassian MCP server supports OAuth 2.1, but the scope of the token still has to be configured on the Atlassian side — do that explicitly rather than handing over a wide-permission token 1.

Stage 2 — Spec generation

The spec-writer reads the JSON from Stage 1 and produces an implementation spec: which files change, the approach, the edge cases, and the open questions. It does not write code.

yaml
# .claude/agents/spec-writer.md
---
name: spec-writer
description: Generate an implementation spec from ticket data
tools:
  - Read
  - Bash
---
Given the ticket JSON from ticket-intake, produce a technical spec.
List affected files (with paths), implementation approach (2-3 paragraphs),
edge cases, and open questions. Use Read to inspect actual files before
listing them. Do not modify anything.

Human gate — Plan approval

This is the most important gate in the pipeline. Catching a misinterpreted acceptance criterion before any code is written costs a 30-second review. Catching it after costs a revert and another CI run. The right mechanism is a 

SubagentStop
hook that exits with code 2 to block the pipeline 5:

json
{
  "hooks": {
    "SubagentStop": [
      {
        "command": "bash -c 'echo Plan ready. APPROVE/REJECT? && read -r ans </dev/tty && [ \"$ans\" = APPROVE ] || exit 2'"
      }
    ]
  }
}

SubagentStop
fires whenever any subagent finishes 5. If you want per-subagent gating — approve the spec but auto-pass test generation — you need conditional logic in the hook command keyed off 
$CLAUDE_PROJECT_DIR
state, or invoke distinct named subagents you can branch on. The current hook model does not support per-subagent name matching directly.

Exit code 2 from a hook blocks Claude from proceeding 5. Exit 0 lets it continue. The operator sees the spec, types APPROVE or REJECT, and the pipeline halts or continues.

Stage 3 — Implementation

The implementer subagent receives the approved spec and writes the code. Scope is strictly bounded to the files listed in the spec.

yaml
# .claude/agents/implementer.md
---
name: implementer
description: Implement changes per the approved spec
tools:
  - Read
  - Edit
  - Write
  - Bash
---
Implement the spec. Modify only the files listed. Do not refactor outside
the specified scope. Run the existing test suite after implementation
and report results.

Stage 4 — Test generation

A separate subagent in its own context. Keeping test generation isolated from implementation avoids the failure mode where the same model that wrote the code also writes tests encoding the same assumptions — a common source of tests that pass while the behavior is wrong.

Stage 5 — PR creation

The GitHub MCP server exposes:

  • create_pull_request
  • actions_list
  • get_job_logs
  • issue_read
  • issue_write 9

The pr-creator subagent commits the changes, opens a draft PR, and polls Actions status until CI completes:

yaml
# .claude/agents/pr-creator.md
---
name: pr-creator
description: Open a PR and verify CI passes
tools:
  - Bash
  - mcp__github__create_pull_request
  - mcp__github__actions_list
  - mcp__github__get_job_logs
---
Commit changes on a feature branch. Open a draft PR with title from the
ticket, body listing acceptance criteria as a checklist, and a link back
to the Jira ticket. Poll actions_list until CI completes. If failures,
fetch logs via get_job_logs and surface them.

Merge approval is enforced by GitHub branch protection rules — not by Claude. This is deliberate: quality gates (CI must pass) and human-approval gates (human must click merge) are architecturally distinct. Branch protection enforces both deterministically. Claude does not need to ask whether it has permission to merge.

For the CI side, 

claude-code-action@v1
runs inside a workflow triggered on PR open:

yaml
# .github/workflows/claude-review.yml
name: Claude PR Review
on:
  pull_request:
    types: [opened, synchronize]
jobs:
  claude-review:
    runs-on: ubuntu-latest
    steps:
      - uses: anthropics/claude-code-action@v1
        with:
          claude_code_oauth_token: ${{ secrets.CLAUDE_CODE_OAUTH_TOKEN }}
          claude_args: "--max-turns 10"

--max-turns 10
is a real, documented flag that bounds the agent loop — appropriate for a CI review job where you want predictable, finite behavior 10.

Stage 6 — Security assessment

/security-review
is a built-in command discoverable via the Skill tool, confirmed in the Week 16 changelog covering versions v2.1.105–v2.1.113 (April 13–17, 2026) 7 and documented in the Skills reference 6. You invoke it from inside a security subagent:

yaml
# .claude/agents/security-reviewer.md
---
name: security-reviewer
description: Run a security review on the changes in the current branch
---
Invoke the /security-review skill on the changes introduced by the
implementer in this branch. Focus on: authentication, input validation,
secrets handling, dependency vulnerabilities. Output a structured finding
list with severity, location, and recommended fix.

For a deeper pass, 

/ultrareview
(research preview, requires v2.1.86+) launches a multi-agent fleet that runs in a remote sandbox and independently verifies each finding before reporting 8. The verification step matters — single-pass model output produces false positives that erode trust in the security stage. Cost is documented: three free runs, then $5–$20 per run, with a 5–10 minute runtime 8. Whether that cost is worth it depends on the change — overkill for a CSS refactor, appropriate before shipping an authentication change.

The security stage runs after the PR exists but before deploy. Findings can be filed as comments on the PR or as new Jira tickets via the Atlassian MCP server, closing the loop.

Stage 7 — Deploy with a PreToolUse gate

There is no single primary source documenting an integrated Jira-to-deploy Claude Code reference architecture. The deploy step is a composition pattern: your existing deploy CLI is gated by a 

PreToolUse
hook on 
Bash
5.

Common deploy commands include:

  • vercel deploy --prebuilt
  • aws cloudformation deploy
  • fly deploy

PreToolUse
is the only hook type that can block a tool call before it executes 5.

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "bash -c 'cmd=$(jq -r .tool_input.command); if echo \"$cmd\" | grep -qE \"(deploy|release|push.*prod)\"; then echo \"DEPLOY DETECTED: $cmd\" >&2 && read -p \"Confirm? [yes/no]: \" ans </dev/tty && [ \"$ans\" = yes ] || exit 2; fi'"
      }
    ]
  }
}

The hook reads the tool input as JSON from stdin via 

jq
(the documented input mechanism for command hooks 5), extracts the Bash command, pattern-matches against deploy verbs, and surfaces the exact command to the operator. Exit code 2 cancels the tool call without terminating the session. Any non-deploy Bash call passes through unchanged.

Where the human gates go, and why

Three gates, three different jobs:

Gate 1 — Plan approval (Stage 2, 

SubagentStop
). Implementation direction errors are expensive. A 30-second plan review is the cheapest correction in the pipeline. The hook blocks deterministically; it does not rely on Claude politely asking 5.

Gate 2 — Merge approval (Stage 5, branch protection). Enforced by GitHub infrastructure, not by Claude Code. This separation is the right architecture: the gate that should be hardest to bypass is the one furthest from the model.

Gate 3 — Deploy confirmation (Stage 7, 

PreToolUse
). Deploy commands are irreversible on short timescales. The 
PreToolUse
hook surfaces the exact command and requires explicit confirmation before anything hits production 5.

Why subagents and not one big Skill

A single large Skill could describe the workflow as a checklist, but it would still run in one shared context. That means ticket intake, spec generation, implementation, tests, PR review, and deploy all compete for the same attention and memory budget.

Subagents turn those steps into isolated execution units. Each stage gets its own context window, its own system prompt, its own tool allowlist, and a structured handoff to the next stage 3. That separation reduces context pollution, makes permission boundaries real instead of aspirational, and gives human approval points a clean place to stop the pipeline.

This is the deeper architectural reason the pipeline is composed from subagents rather than written as one giant instruction block. A Skill can explain the process; subagents enforce the separation that makes the process safe.

Caveats and open questions

Agent teams remain experimental. As of the Week 16 changelog they require v2.1.32+ and are flagged experimental 47. The shared task list, 

TeammateIdle
and 
TaskCompleted
hooks, plan-approval workflow — all of this can change. Do not build production infrastructure on the team coordination primitives yet.

/ultrareview
is metered. Three free runs, then $5–$20 per run 8. At $20/run with ten deploys a week, that is $200/week in security review costs. Budget this explicitly.

The deploy stage is composition, not integration. The pattern here is derivable from the hooks docs and is architecturally sound, but no published reference architecture documents the full Jira-to-deploy Claude Code pipeline end-to-end. Treat Stage 7 as a design pattern, not a documented integration.

Atlassian OAuth scope management is your problem. The MCP server handles the auth flow, but you still configure which Jira projects and issue types the integration can touch 1. An overly permissive scope on a company-wide Jira is a real risk.

SubagentStop
fires for every subagent. Selective gating requires conditional logic in the hook command or per-stage subagent definitions you can branch on 5. There is no built-in matcher syntax for subagent names.

PostToolUse
cannot block. If you use it for test feedback rather than a 
SubagentStop
hook, a model that decides to ignore the failure can keep writing code. For hard enforcement, use 
SubagentStop
with exit 2 5.

Ticket quality is the ceiling. Vague acceptance criteria produce vague specs produce vague implementations. The pipeline amplifies ticket quality in both directions.
