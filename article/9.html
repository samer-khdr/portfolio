Multi-Agent vs Single-Agent: Concrete Engineering Tradeoffs

Samer Alkhodary
•6 min read•6 topics

The instinct to decompose a struggling agent into a multi-agent system is common and often premature. Here are the actual token costs, failure modes, and architectural tradeoffs across single-agent, supervisor, handoff, and swarm patterns — grounded in published data, not framework marketing.

#AI Agents#Multi-Agent#Architecture#LangGraph#Token Cost#Agent Orchestration

TL;DR

  • A single agent with many tools is the right default. Multi-agent is a scaling tool, not an architecture upgrade.
  • Multi-agent systems use roughly 15x more tokens than standard chat interactions; single-agent uses roughly 4x. That cost spread is real and compounds fast at scale 1.
  • Parallelism is the clearest win for multi-agent: Anthropic cut research time by up to 90% on complex queries by running subagents in parallel 1. But parallel agents share state poorly and produce conflicting outputs when not designed carefully 2.
  • Before going multi-agent: if your task fits one agent with 3-10 tool calls, it probably should 1.

Why this matters

The default instinct when an agent starts underperforming is to decompose it — spin up a supervisor, hand work off to specialists, run things in parallel. Sometimes that is the right call. But multi-agent systems introduce coordination overhead, token costs, and failure modes that single-agent systems sidestep entirely. If you are building your next agent and choosing an architecture, the decision deserves a concrete engineering analysis, not a pattern selection from a framework README.

The patterns and what they actually cost

§Single agent + tools

One LLM, one context window, a set of registered tools. The agent decides when to call which tool, synthesizes results, and responds. This is the baseline.

What you get: full shared context at all times, deterministic call chains if you want them, easy tracing, and the lowest token overhead of any approach. Debugging means reading one log. State sharing is trivial — it is all in one context.

What breaks it: too many unrelated instructions or tools crammed into a single context. Arruda's Shopify team observed this directly — monolithic prompts where "you have too many unrelated tokens, too many instructions, the LLM gets lost, and the result was very poor" 6. At that point, decomposition is not premature optimization; it is necessary.

Anthropic's internal decision rule is concrete: simple fact-finding maps to one agent with 3-10 tool calls 1. If your task fits that description, the added complexity of any multi-agent pattern is overhead you are paying for nothing.

§Supervisor / manager pattern

A central LLM owns the conversation and delegates to specialists, either via Agent.as_tool() (OpenAI Agents SDK) or tool-call-based routing (LangGraph). The supervisor sees all inputs and outputs; subagents do bounded subtasks and return results.

This is the most common multi-agent pattern and has a real use case: parallel execution of independent subtasks. Anthropic's research system runs lead agents that spawn 3-5 subagents in parallel, each using 3+ tools concurrently, cutting research time by up to 90% for complex queries 1. Shopify decomposed a theme review process that took 22 hours as a monolith down to 7-20 minutes by assigning each review criterion to a separate specialized agent 6.

The token cost for a multi-agent system sits at roughly 15x the baseline chat interaction 1. A single agent runs roughly 4x baseline. That 3-4x gap between single-agent and multi-agent is the cost of coordination, repeated context injection, and subagent scaffolding — before you account for supervisor round trips.

One concrete efficiency lever: Lin et al. found that a SupervisorAgent framework reduced token consumption in the Smolagents framework by 29.68% on the GAIA benchmark without compromising success rates, validated across five additional benchmarks 5. The gain comes from runtime supervision that filters redundant observations and corrects errors before they propagate. Naive supervisor implementations waste tokens, and there is real room to recover some of that cost with smarter orchestration.

One note on tooling: LangChain's own README for the langgraph-supervisor library now reads: "We now recommend using the supervisor pattern directly via tools rather than this library for most use cases. The tool-calling approach gives you more control over context engineering." 4 The library's own maintainers walking back the abstraction in favor of more explicit tool calls is worth paying attention to.

§Handoff / swarm pattern

A triage agent routes the conversation to a specialist, which becomes the active agent. Unlike agents-as-tools, control transfers — the specialist is not called and returned from; it takes over the remainder of the turn.

In the OpenAI Agents SDK, the receiving agent sees the full prior conversation history by default. You can filter this via input filters or the (beta) nested handoff history option, which collapses prior context into a summarized message 3. The tradeoff is direct: full history means the specialist has complete context but pays for it in tokens on every call; filtered history is cheaper but risks losing relevant detail.

The handoffs documentation flags a guardrail gap worth knowing: "Handoffs stay within a single run. Input guardrails still apply only to the first agent in the chain, and output guardrails only to the agent that produces the final output." 3 If your safety or compliance setup assumes guardrail coverage at every step, a handoff chain breaks that assumption silently.

The SDK docs also make a point that applies across all these patterns: code-based orchestration, where routing logic lives in your code rather than in an LLM, "makes tasks more deterministic and predictable, in terms of speed, cost and performance" 3. For triage workflows where routing rules are finite and known, hard-coded routing beats an LLM deciding where to hand off — it is cheaper, faster, and you can test it.

§Hierarchical and nested patterns

Supervisors of supervisors. Token cost compounds at every level, and debugging a failure requires tracing the full call tree. None of the sources I have give separately benchmarked cost figures for hierarchical vs. flat multi-agent topologies — I am not going to invent a comparison that does not exist in the data. What the sources do show: Anthropic's research system uses explicit effort ceilings (simple = 1 agent, complex = 10+ subagents) rather than treating deeper hierarchy as the default 1. The pattern is real, but the recommendation from practitioners is to reach for it only when task complexity genuinely demands it.

The parallel execution problem

The strongest argument for multi-agent is latency reduction, not quality improvement. Token-for-token, a multi-agent system running subagents in parallel can finish faster than a single agent working sequentially, even though it costs more. Anthropic's data is specific: parallel execution cut research time by up to 90% for complex queries, and token usage explains 80% of performance variance on BrowseComp 1. Running subagents in parallel is how you scale token budget without linearly scaling wall time.

But parallel subagents share state poorly. Yan describes the failure mode clearly: two parallel subagents building a Flappy Bird clone — one produces a Super Mario Bros background, the other an incompatible bird asset, and the downstream agent cannot reconcile them 2. This is not a contrived edge case; it is the natural outcome of agents that make implicit decisions without shared context. Yan's principle applies directly: "Actions carry implicit decisions, and conflicting decisions carry bad results." 2

Claude Code's architectural response to this is deliberate: it never runs work in parallel with a subtask agent, and subtask agents are typically only tasked with answering questions, not writing code 2. Sequential is slower but sidesteps state conflict entirely.

Caveats and open questions

No clean per-pattern cost breakdown exists in the public literature. The 15x vs. 4x token figures from Anthropic 1 are high-level — multi-agent vs. single-agent. There is no published, source-backed number comparing supervisor vs. swarm vs. hierarchical token overhead. If that breakdown matters to your decision, benchmark your own workload.

Latency numbers by pattern are not in the sources. Anthropic's 90% research time reduction 1 is specific to parallel execution in their research system. Shopify's 22-hours-to-minutes improvement 6 compares a broken monolithic prompt against a well-designed multi-agent system — not a clean A/B of architectures. Both are real, neither generalizes cleanly to your workload.

Debugging is qualitatively harder, not quantified. Arruda's mitigation was running agents in-process via Ruby fibers rather than over the network, making mid-event tracking straightforward 6. That is a useful pattern — keeping agents in-process during development preserves observability. But the sources do not quantify the debugging overhead as you add agents. It gets worse; by how much depends on your tooling.

Benchmark provenance matters. Anthropic's 90.2% multi-agent improvement is on Anthropic's internal research evaluation 1. Lin et al.'s 29.68% token reduction is on GAIA 5. These numbers are real, but they describe specific task distributions. Treat them as directional signals, not guarantees.

Guardrail coverage gaps in handoff chains. If your safety requirements assume input and output guardrails at every step, the OpenAI Agents SDK handoff architecture requires explicit re-examination — guardrails only fire at the first and last agents in the chain 3.
